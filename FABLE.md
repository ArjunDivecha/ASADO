## Assessment

ASADO is a genuinely serious research platform, and the code reflects that. The engineering is
strong where it counts: the skeptic harness (`scripts/harness/evaluate_signal.py`) has an
explicit, empirically-justified forward-return ban (`:140-147`, enforced at `:599-601`), PIT
embargo, NeweyŌĆōWest t, deflated Sharpe, and a cost grid; the warehouse cleanly separates the
recreate-on-every-build main DB from the durable loop DB; and the DuckDB lock outage of
2026-07-02/03 was met with a real fix (`scripts/duckdb_lock_guard.py::guarded_connect()`) plus a
written postmortem in `AGENTS.md`. The `AGENTS.md` / experiment-sandbox governance is unusually
disciplined for a solo research repo. This is not a codebase that needs rescuing.

The correctness risk is not in the rules ŌĆö it is that **the most important rules are enforced by
convention and duplication rather than by a single gate**, and the repo has almost no CI
(`.github/workflows/` contains only `openwiki-update.yml`). Three concrete instances:

- **The forward-return blacklist is re-implemented independently.** The canonical set lives at
  `scripts/harness/evaluate_signal.py:146`; the same "LOOKAHEAD TRAP" is re-derived by hand at
  `scripts/loop/triptych_kernel.py:84-90`. Any new signal path that forgets to import the
  canonical set (or diverges its regex) can register a target as a predictor and no gate stops it.
- **Isolated-table exclusion is a house rule, not a check.** `feature_panel`/`unified_panel` are
  built in `scripts/build_normalized_panel.py:380-395`; nothing fails the build if a future edit
  unions `factor_returns`, `ff_factors`, or the JST corpus into those views ŌĆö the cycle/broadcast
  guarantees rely entirely on the author remembering.
- **`feature_panel` carries 43 countries, not 34.** Per the R3 audit note at
  `build_normalized_panel.py:28-32`, non-T2 names leaked in and `feature_panel_t2` was added as
  the "documented default," but the default is documentation, not enforcement ŌĆö every consumer
  must still re-filter by hand, and a new consumer that reads raw `feature_panel` silently mixes
  Austria/Greece into a 34-country cross-section.

None of these are bugs today. They are the class of latent risk Divecha exists to close: turn the
vigilance into a command. The rest is polish (state-log drift, blacklist de-duplication).

## Prioritized improvements

### P0 ŌĆö Central deterministic leakage/isolation guard
**What & why:** collapse the two crown-jewel rules (forward-return ban + isolated-table
exclusion) into one guard that exits nonzero and names offenders, and wire it into CI. Today they
are enforced across Ōēź8 files by convention (`evaluate_signal.py:146` vs `triptych_kernel.py:84`)
with no gate. This is the highest leverage in the repo: it protects every future experiment and
collector.
- **Bet** ŌĆö *if* the guard runs against `config/family_registry.yaml` and the
  `feature_panel`/`unified_panel` view SQL, *then* it exits nonzero naming any forward-return
  variable registered as a signal or any isolated table referenced by a panel view, and exits 0
  otherwise; *observable*: exit 0 on the clean repo, nonzero on each seeded fixture with the
  offender printed.
- **Invariants:** INV1 no NMRet-family variable is a registered signal; INV2 no isolated table
  (`ff_factors`, JST, `factor_returns`, `factor_top20_membership`, `country_factor_attribution`)
  is in a panel view; INV3 the guard is read-only and holds no live-DB connection.
- **Gate intents:** clean repo ŌåÆ exit 0; seeded forward-return registration ŌåÆ nonzero; seeded
  isolated-table-in-view ŌåÆ nonzero; `git diff --name-only` empty after a run; `pytest tests/` and
  `scripts/qa/validate_returns_first.py` stay green.
- **Effort:** ~0.5 day. (Full contract below.)

### P1 ŌĆö Enforce `feature_panel_t2` as the analysis default
**What & why:** stop the 43-vs-34 country contamination (`build_normalized_panel.py:28-32`) from
recurring by making the T2 restriction enforceable, not advisory.
- **Bet** ŌĆö *if* a warehouse consumer reads a cross-sectional panel for analysis, *then* it
  resolves to the 34 T2 names (via `feature_panel_t2`) unless it explicitly opts into raw
  `feature_panel`; *observable*: a test that asserts `SELECT COUNT(DISTINCT country) FROM
  feature_panel_t2` == 34 and that `db_bridge`'s default panel accessor returns only T2 names.
- **Invariants:** default cross-section = exactly the 34 T2 names; raw `feature_panel` reachable
  only through an explicit flag.
- **Gate intents:** count-distinct-country test on the default accessor returns 34; grep/lint that
  no loop builder selects raw `feature_panel` without the opt-in.
- **Effort:** ~0.5 day.

### P1 ŌĆö Assert every pipeline DB-open routes through `guarded_connect()`
**What & why:** the lock guard exists (`duckdb_lock_guard.py`) but adoption is by convention; the
2026-07-02/03 outage was exactly a non-guarded holder. Make the routing a checked invariant so it
cannot regress.
- **Bet** ŌĆö *if* any script under `scripts/` opens either live DuckDB, *then* it does so via
  `guarded_connect()`; *observable*: a static-scan test that greps for `duckdb.connect(` outside
  the guard module and fails if any pipeline path bypasses it.
- **Invariants:** no direct `duckdb.connect()` to the two live DB paths outside `duckdb_lock_guard`.
- **Gate intents:** the scan returns empty (exit 0); a seeded direct-connect fixture makes it fail.
- **Effort:** ~0.5 day.

### P2 ŌĆö Single source of truth for the forward-return blacklist
**What & why:** remove the duplication that P0 defends against, at the root ŌĆö one importable
constant/regex used by both `evaluate_signal.py:146` and `triptych_kernel.py:84-90`.
- **Bet** ŌĆö *if* the blacklist is defined once and imported, *then* both call sites reference the
  same object; *observable*: a test asserting `triptych_kernel` and `evaluate_signal` import the
  same blacklist symbol and the module-local copies are gone.
- **Invariants:** exactly one definition of the forward-return set/regex in the codebase.
- **Gate intents:** grep finds a single definition; the P0 guard consumes it.
- **Effort:** ~1-2 hours.

### P2 ŌĆö Lightweight state-of-play freshness check
**What & why:** `llmchat.md` is the cross-agent context log but was ~20 commits stale at review
time; a fresh agent trusting it gets a 4-day-old picture.
- **Bet** ŌĆö *if* HEAD is more than N commits or M days ahead of the last `llmchat.md` entry,
  *then* a check warns (non-fatal); *observable*: the check prints the gap and its exit code is
  informational.
- **Invariants:** the check never blocks the pipeline (advisory only).
- **Gate intents:** on a stale log the check reports the commit/day gap; on a fresh log it is silent.
- **Effort:** ~1 hour.

## Context

ASADO (`/Users/arjundivecha/Dropbox/AAA Backup/A Working/ASADO`) is a macro-research
platform whose product is *trustworthy falsification* of candidate trading signals. Two
correctness rules are the crown jewels, and today both are enforced only by convention and
human vigilance (they are described in `AGENTS.md` and re-implemented ad hoc across at least
eight scripts, e.g. `scripts/harness/evaluate_signal.py` and `scripts/loop/triptych_kernel.py`):

1. **Forward-return blacklist.** The T2 `NMRet` variables ŌĆö `1MRet`, `3MRet`, `6MRet`,
   `12MRet` and their daily analogs ŌĆö are the optimizer's prediction *targets* (future
   returns), not trailing momentum. Registering one as a predictor is direct look-ahead
   leakage. Harness registrations live in `config/family_registry.yaml`.
2. **Isolated-table exclusion.** `ff_factors` (8 regional Fama-French series), the JST
   Macrohistory calibration corpus, and the optimizer outputs (`factor_returns`,
   `factor_top20_membership`, `country_factor_attribution`) must never be unioned into the
   query-facing views `feature_panel` or `unified_panel`. The optimizer-output exclusion
   breaks the optimizer input/output cycle; the FF/JST exclusion prevents region-level series
   being broadcast/tiled across the 34 countries. These views are built by
   `scripts/build_normalized_panel.py` and consumed by `scripts/db_bridge.py` and
   `scripts/asado_mcp_server.py`.

This contract introduces a single deterministic guard (a coding model should discover the
exact path in Build Mode; the natural home is `scripts/qa/`, alongside the existing
`scripts/qa/validate_returns_first.py`) that mechanically re-checks both rules and exits
nonzero, naming offenders, whenever either is violated. It reads files and view definitions
only; it never writes, and never holds a connection to the live DuckDB files.

Terms: *harness signal* = a variable registered as a predictor family in
`config/family_registry.yaml`; *panel view* = `feature_panel` or `unified_panel`;
*isolated table* = one of the five tables named in INV2; *fixture* = a throwaway copy of a
registration file or view-SQL string with a violation injected, used only to prove the guard
fails when it should.

## Build Loop vs Product Loop

The **build loop** is fully machine-checkable: the guard passes (exit 0) on the current
repo (G1), fails (exit nonzero) on a seeded forward-return registration (G2) and on a seeded
isolated-table reference in a panel view (G3), is side-effect-free (G4), and leaves the
existing suite green (G5). These gates prove the guard reliably distinguishes a clean state
from the two named violation classes.

The **product loop** is the real-world bet: that ASADO never again lets a leaked or
forward-looking feature reach a live trading decision, and that the discipline holds across
months of new experiments and collectors. That outcome can only be judged by live operation
and periodic audit, not by this guard alone ŌĆö the guard covers the two known leakage classes,
not every future one. The coding model **may not claim** the product bet is satisfied merely
because the build-loop gates pass; passing gates prove the guard works, not that the platform
is leak-free forever.
