## 0. Read this first — the cautionary tale this PRD is designed around

`ASADO/regime/` already tested a closely related hypothesis (call it H2-global: does T2 factor IC vary by regime) using ONE global US-macro regime label applied to all 34 countries at once. **There are two conflicting reports for the exact same test**: `regime2.md` says it FAILED (0/52 factors significant after FDR, α=0.10 per the text). `regime/regime.md` — generated by the code currently checked into `regime/src/` — says it PASSED (21/52 significant). The reason: `ASADO/regime_loop/` was an autonomous loop explicitly built to search over regime-threshold definitions, IC methods, backtest construction, and FDR alpha/method until the hypothesis passed (`regime_loop/strategy.md`, `program.md` literally list "relax FDR correction," "modify significance threshold," "try bootstrap-based significance testing" as tuning knobs). It crashed mid-run right after landing on the config that flipped the verdict, and that tuned config is what's in the repo today. Separately, `ic_analysis.py`'s FDR alpha in code is 0.20, while both reports' text says "α=0.10" — a second, unresolved inconsistency.

**This PRD exists specifically to not repeat that failure mode.** Every threshold below is frozen before any test runs. No code or report may change a regime definition, factor list, or FDR choice after seeing a preliminary p-value — a new idea gets a new versioned PRD, not a silent edit to this one.

**Also read `ASADO/momentum_fragility/results/RESULTS.md` and `ASADO/regime_ew/results/RESULTS.md`** (both from earlier this session) — both are honest negative results on adjacent hypotheses, written in the style this PRD's own RESULTS.md must match.

## 1. Objective

Two objectives, kept explicitly separate:

**(A) Descriptive only, not predictive:** classify each of the 34 T2 countries, each month, into Growth / Recession / Neutral based on that country's own real economic activity (industrial production primarily, GDP growth as a separate exploratory check). No backtest, no gates — this is a labeling tool, useful on its own for dashboards/context.

**(B) The decisive test:** conditioned on each country's OWN regime state (not a global label), does any of T2's ~83 factors show a significantly different return-predictive relationship (IC) depending on regime? I.e., "should we pick different factors for a country depending on whether it's in growth, recession, or neutral." This is an interaction/conditioning question, not a claim that regime itself predicts returns (that was already tested, differently, in `regime_ew`, and failed).

**Not the objective:** predicting the regime, predicting returns from the regime directly, or building a production classifier. Objective (A) has no pass/fail gate — it's just a labeled panel. Objective (B) has hard pre-registered gates below.

## 2. Data

### 2.1 Classifier inputs (Objective A)

**Primary series: monthly Industrial Production YoY.** Not in ASADO today — pull via Bloomberg (OpusBloomberg, `bbg.ref`/`bbg.hist`). Live-verified this session: US (`IP CHNG Index` MoM, `IP YOY Index` YoY), Germany (`GRIPIYOY Index`), Japan (`JNIPYOY Index`), China (`CHVAIOY Index`), Korea (`KOIPIYOY Index`), Eurozone (`EUIPEMUY Index`). The remaining ~28 T2 countries need per-country ticker discovery (test 1-2 candidates live before any batch pull, per this codebase's Bloomberg discipline — see `bloomberg-skill/references/lessons.md`'s ticker-verification rules). Expect real attrition — not all 34 will resolve; document exactly which do and don't, this determines the test's actual country count.

Mapping notes: `NASDAQ` and `US SmallCap` (pseudo-countries in `T2_COUNTRIES`) both use the US IP series. `ChinaA` and `ChinaH` both use mainland China's IP series (same underlying economy, `Hong Kong` is separate and needs its own ticker).

**Secondary series (exploratory only, not primary): annual GDP growth**, already in the warehouse (`t2_raw`/`t2_master` variable `GDP`, or `imf_factors.IMF_WEO_GDP_Growth`). **Do not blend IP and GDP into one classifier.** They move at different frequencies and mixing them per-country would change the measurement instrument country-by-country, risking a manufactured "regime effect." Run the GDP-based classification as a separate, clearly labeled robustness/exploratory comparison on the full 34-country universe, reported alongside but never pooled into the primary IP-based test's FDR family.

**Point-in-time discipline (hard requirement):** both `t2_raw.GDP` and `imf_factors.IMF_WEO_GDP_Growth` already contain forward-dated values (2026 in t2_master, up to 2031 in imf_factors) — i.e., these look like consensus/forecast vintages, not clean point-in-time actuals. Apply an explicit publication lag (do not use a calendar year's GDP figure until well after that year ended — e.g., require the value to be at least 6 months old from its `date` before using it) for the GDP exploratory check. IP data from Bloomberg's `... Index` release series is generally reported with a much shorter, known lag (weeks, not months) — confirm and apply the correct lag per country if available via `ACTUAL_RELEASE`/publication-date fields; if unavailable, apply a conservative 6-8 week lag by default.

### 2.2 Classifier construction

Per country, per month, using ONLY that country's own trailing history up to and including that month (expanding window, minimum 24 months before any label is assigned — same discipline as `momentum_fragility` and `regime_ew`): compute the percentile rank of the current IP YoY value within the country's own trailing history. **Bottom 30% → Recession. Top 30% → Growth. Middle 40% → Neutral.** These percentiles are frozen now; do not tune them after seeing results. Adapt the point-in-time expanding-quantile pattern from `T2 Factor Timing Fuzzy/Step Fifteen Market Regime Analysis.py` (`vol_history = market_vol.loc[:date].dropna(); low = vol_history.quantile(0.25)` etc.) — this is new glue code per-country, not an existing function to call directly (that file operates on one market-wide series today).

### 2.3 Factor panel (Objective B)

All ~83 T2 factors (`Step Factor Categories.xlsx`, both `_CS` and `_TS` where present) — not just the 52-factor `_CS` subset the original H2 pipeline happened to use. Reuse directly, unmodified: `ASADO/regime/src/data_loader.py`'s `load_factor_panel()`, `load_country_returns()`, `build_forward_returns()`. These build a `date, country, factor, signal, ret` panel (factor value lagged 1 month per `(country, factor)`, joined to forward return) — this part of the original H2 pipeline is pure data plumbing and is NOT tainted by the tuning-loop history; only the regime-tagging and IC/FDR test logic downstream of it are.

## 3. The statistical test (Objective B) — read carefully, this is the part that needs care

**Why a naive approach is wrong:** `_CS` factors are cross-sectionally z-scored specifically so their predictive power (IC) measures *relative country ranking within that month*, netting out common monthly shocks. Two naive designs both break this:
- Pooling raw `(country, month)` signal/return pairs globally and bucketing by each country's own regime reintroduces exactly the common-monthly-shock noise cross-sectional ranking exists to remove (a month where all countries rally together injects spurious correlation).
- A plain interaction regression on raw (unranked) values has the same problem.

**Correct design:** for each date, rank BOTH the factor signal and the forward return across that date's full country cross-section (the same transform `spearman_ic` already does — reuse it, don't reimplement). Stack these within-date ranks into a panel, tag each `(country, date)` row with that country's own regime label (from §2.2). For each factor, fit:

```
rank_ret ~ rank_signal + rank_signal:regime_dummy_Growth + rank_signal:regime_dummy_Recession
```
(Neutral as the omitted baseline category; two interaction terms for the other two regimes.)

Test the **joint significance of the two interaction terms** (a Wald/F test) — this is the direct test of "does this factor's return-predictive slope differ by regime."

**Inference:** use a **wild cluster bootstrap over countries** for the interaction terms' p-values, not plain asymptotic cluster-robust standard errors. With an expected ~15-25 countries having usable IP data (post-attrition from §2.1), that's below the ~30-50 cluster rule-of-thumb where standard asymptotic cluster SEs are reliable.

**Multiple testing:** Benjamini-Hochberg FDR across the ~83 factors' joint-significance p-values, at **α=0.10** (a deliberate, explicit choice — not inherited from `ic_analysis.py`'s conflicting 0.10-vs-0.20 code/text mismatch; document this choice in the code and RESULTS.md).

**Minimum cell size:** for a factor/regime combination to be included, require at least 20 country-month observations in that regime bucket (regimes are trending/autocorrelated — multi-month spells mean effective N is much smaller than row count; also expect global-cycle synchronization, e.g. 2008-09, 2020, to create near-empty buckets during crises — report this, don't paper over it or silently drop it).

**Pre-registered null-result statement (write this now, before running anything):** if fewer than 8 of the ~83 factors (~10%, roughly matching the FDR alpha itself) clear FDR-corrected significance on the joint interaction test, conclude there is no genuine per-country regime-conditional factor-selection edge in this data, and stop. Do not sweep the classifier percentile split, the FDR alpha, or the minimum-cell-size threshold looking for a passing configuration afterward — that is the exact failure mode this PRD exists to avoid. `regime2.md`'s own root-cause argument (T2 factors' edge is relative cross-sectional ranking, largely orthogonal to macro state) is a legitimate reason to expect this outcome; it is a valid, useful result, not a bug to fix.

## 4. Robustness checks (run regardless of outcome, report honestly)

- **Placebo:** shuffle each country's regime-label time series independently (breaking the true country-date regime alignment while preserving each regime's marginal frequency), rerun the full test. The number of factors clearing FDR must collapse toward the false-discovery rate itself (i.e., ~α × 83 factors by chance). If the placebo also shows a similarly large number of "significant" factors, the real result is a pipeline artifact.
- **Garden-of-forking-paths accounting:** explicitly state in RESULTS.md that the classifier design itself involved researcher choices (IP vs. GDP, 30/30 vs. some other split) that constitute additional implicit tests beyond the 83 factors — do not present the FDR correction as if it accounts for every choice made to get here.
- **Git commit stamp:** RESULTS.md must record the git commit hash of the code that produced it (guards against the exact `regime2.md`-vs-`regime.md` silent-overwrite problem).

## 6. Decision framework

| Outcome | Interpretation | Next action |
|---|---|---|
| ≥8 factors clear FDR AND placebo shows near-zero | Genuine regime-conditional factor edge, per-country | Consider Phase 2: does regime-conditional factor selection improve backtested performance (new, separately scoped test) |
| <8 factors clear FDR | No genuine edge (pre-registered null) | Stop. Write up honestly, in the `regime2.md`/`regime_ew` tradition |
| ≥8 factors clear FDR but placebo ALSO shows a similar count | Artifact, not a real effect | Stop. Document exactly which check caught it |
| Too few countries have usable IP data (say, <10) to run a meaningful test | Data insufficiency, not a hypothesis failure | Report this distinctly from a null result — it's a different conclusion (untested, not tested-and-failed) |
