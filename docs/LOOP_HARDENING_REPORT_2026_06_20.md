# ASADO Alpha-Hunting Loop ‚Äî Prioritized Hardening Report

**Branch:** `claude/nightwatch-06-20-failures-37d4lv` ¬∑ **Date:** 2026-06-20 ¬∑ **Scope:** governance layer (`scripts/loop/`), nightly orchestrators (`scripts/daily_update.py`, `scripts/loop/loop_daily_job.py`), external-infra (Bloomberg/Parallels, Neo4j, DuckDB, Dropbox, launchd).

## 1. Executive summary

The loop's governance layer is **conceptually sound but trust-fragile at its silent-failure boundaries.** Its stated contract ‚Äî "knows what it doesn't know," any uncomputed dimension goes BLIND not green ‚Äî is defeated by two confirmed paths where the system reports a confident GREEN/OK while having verified nothing: an empty/truncated `governance_contract.yaml` greens the whole scorecard (`gov-empty-contract-green`, **CRITICAL**), and a git failure greens `config_guard` and blanks the repo-dirty header (`gov-git-empty-on-failure-silent`, **HIGH**). These directly invert the subsystem's purpose.

The larger systemic exposure is **unbounded subprocess execution.** Neither nightly orchestrator nor the governance tail applies a single `timeout=` to any `subprocess.run`. A wedged Bloomberg pull, an up-but-unresponsive Neo4j bolt, or a Dropbox filesystem stall hangs the entire pipeline indefinitely ‚Äî worse than a clean fail, because the foreground launchd wrapper never returns, the heartbeat never writes, and the only backstop is a watchdog hours later. This is the single highest-leverage hardening: **the loop has retry logic everywhere and interrupt logic nowhere.**

Durability is mostly good ‚Äî `run_manifest`/`heartbeat`/`scorecard.json` use atomic temp+rename ‚Äî but two write paths break the house pattern: the **predmkt archive COPY overwrites the only rebuild-proof history store in place** (`predmkt-archive-non-atomic-corruption`, **HIGH**), and the **brief prepend is a non-atomic direct write with fragile marker-splitting** (`gov-brief-prepend-corrupt-marker`, **MEDIUM**).

Finally, the entire system **trusts the Dropbox-synced filesystem as if it were local** ‚Äî mtime-based staleness, glob-by-newest brief selection, and bare `exists()/st_size` checks are all mutable by the Dropbox sync agent, producing false-fresh, false-stale, and conflict-copy-selected outcomes. These are mostly low-to-medium severity individually but compound into misattributed alerts that erode operator trust.

Of the 14 applied fixes, **13 are adequate; one (the DuckDB connection retry in `loopdb.py`) is partial** ‚Äî it does not survive the 8‚Äì12-minute monthly `setup_duckdb.py` rebuild window it is most likely to face, blind-catches non-lock errors, and can leak a connection on ATTACH failure that manufactures the very contention it fights.

## 3. Applied fixes that are inadequate / partial

Of the 14 re-checks, 13 returned adequate/null. **One is partial:**

**`loopdb.py` DuckDB connection retry (`loop_connection`, `scripts/loop/loopdb.py:70-82`) ‚Äî PARTIAL.**

The mechanical claim is true: it does 1 initial + 4 retries with 2/4/8/16s backoff and re-raises after exhaustion. But four gaps remain, and they matter precisely in the window it was built for:

1. **Does not survive the realistic main-DB contention window.** `setup_duckdb.py` deletes and recreates `asado.duckdb` over 8‚Äì12 minutes. `MAIN_DB.exists()` passes against the half-written file, the `ATTACH '...' AS asado (READ_ONLY)` at `:75` fails, and 30s of total backoff cannot outwait a 10-minute rebuild ‚Äî it just delays the inevitable raise. **Add:** detect the rebuild window explicitly (e.g. a lock/sentinel file written by `setup_duckdb.py` for its duration, or a longer bounded wait specifically when the failure is an ATTACH-on-main error) and surface a clear "main DB rebuilding" status rather than a hard failure.

2. **Blind `except Exception` retries on non-retryable errors** (`:77`) ‚Äî corrupt DB, bad ATTACH path, version mismatch, disk-full all burn the full 30s before dying. **Add:** classify the exception; only retry on lock/IO errors (`duckdb.IOException` / lock-message match); re-raise deterministic errors immediately.

3. **Connection leak on ATTACH failure** (`:74-75`): if `duckdb.connect()` succeeds but `ATTACH` raises, `con` is never closed; the leaked in-process write handle can cause the *next* attempt's `connect()` to fail with a lock error ‚Äî the retry manufactures its own contention. **Add:** `try/except` around the ATTACH that closes `con` before continuing the retry loop.

4. **Compounds with the orchestrator retry.** A raised exception exits the step subprocess with code 1, which `loop_daily_job._run_step` (`:288-294`) treats as **retryable**, re-running the whole step with its own 10s/20s backoff and re-entering the 30s inner loop each time ‚Äî multiplied wasted wall-time under one persistent lock holder. (Exit 2 is the partial/warning path and is correctly never retried, but `loop_connection` raises ‚Üí exit 1 ‚Üí the fatal+retry treatment.) **Add:** when `loop_connection` exhausts retries against a main-DB-rebuild condition, exit **2** (partial/warning), not 1, so the orchestrator does not pile a second retry layer on top.

## 4. Cross-cutting hardening recommendations

1. **Timeouts on every subprocess ‚Äî with process-group teardown.** No `subprocess.run` in the loop has a `timeout=`. Adopt one helper (`run_bounded(cmd, timeout, ...)`) used everywhere ‚Äî `daily_update.run_step`, `loop_daily_job._run_step`, the governance tail, `_auto_commit_brief`, `_git`, `_brief_committed`. It must `Popen(start_new_session=True)` + `communicate(timeout=)` and `os.killpg(SIGTERM‚ÜíSIGKILL)` on expiry, because the hung children are `conda run` wrappers whose grandchildren (python/bbcomm) otherwise orphan and leak the Bloomberg session. This single change fixes findings #3, #4, #9 and most of #10's hang surface.

2. **Atomic writes everywhere ‚Äî audit every `write_text`/`COPY`/`to_parquet` against the temp+rename house pattern.** Already correct: `run_manifest`, `heartbeat`, `scorecard.json`. Broken: predmkt archive COPY (#5), brief prepend (#6). Sweep for any other in-place writer (`grep -rn "write_text\|to_parquet\|COPY.*TO" scripts/loop scripts/predmkt*`) and convert each to `.tmp` + `os.replace` in the same directory.

3. **A single uniform exit-code contract.** Today rc 0 = ok, rc 2 = partial/warning (not retried), other = fail (retried). This convention is implicit and one place (`loopdb` raising ‚Üí exit 1) violates the spirit by routing a "main DB rebuilding, try later" condition into the fatal+retry path. Document the contract in `governance_contract.yaml`, make every step honor it (timed-out optional step ‚Üí 2, transient-dependency-unavailable ‚Üí 2, genuine failure ‚Üí 1), and have `_run_step` retry **only** rc‚àâ{0,2}.

4. **Dependency-readiness checks, not mere port/TCP probes.** Two confirmed false-readiness paths: `detect_vm_ip` accepts the first TCP-open :8194 with no BLPAPI handshake (#11), and the Neo4j steps use a 3s TCP preflight that passes against an up-but-not-serving bolt. Replace bare TCP probes with a minimal real handshake (one BLPAPI ref; one trivial Cypher with `connection_acquisition_timeout`) before declaring a dependency live. Same principle for Dropbox: a "file exists" check is not a "file is materialized and readable" check (#10).

5. **Alerting hardening.** (a) The pusher must never crash before it pushes ‚Äî guard every `json.loads` in `heartbeat.py` (#7) so a corrupt input becomes the alert, not a silent exit. (b) Alerts should name the **root cause**, not a downstream symptom ‚Äî "Dropbox not materialized" / "git unavailable" / "governance subprocess timed out" rather than "brief missing". (c) Close the false-negative governance paths so the scorecard cannot report GREEN/OK while verifying nothing (#1, #2) ‚Äî these are the alerts most worth trusting and currently the easiest to silence. (d) Consider having `heartbeat` read the scorecard `overall` (it currently does not) so a governance RED actually reaches a human, then widen the watchdog tolerance + caffeinate the plists (#17) to remove false pages.
